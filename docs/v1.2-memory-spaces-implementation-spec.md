# Recollectium v1.2 Memory Spaces Implementation Spec

**Status:** draft for implementation

**Target release:** v1.2

**Goal:** Make memory spaces a first-class storage-routing concept across Recollectium Core, replacing normal-operation database-path routing with logical memory-space keys that resolve to one SQLite database per memory space inside a configured database folder.

**Architecture summary:** Recollectium Core should own a configured database folder. Each memory-space key resolves through one central resolver to a safe SQLite database file under that folder. All public surfaces should accept an optional memory-space key and default to the default memory space when omitted. Normal memory operations must not accept arbitrary database paths.

**Non-goal:** This spec does not implement Hermes integration. It prepares Core so a Hermes memory provider can resolve a profile memory-space key once at provider initialization and send that key on every Recollectium request later.

---

## 1. Product decision summary

### Locked decisions

1. **Storage boundary:** One configured database folder, not one configured database file.
2. **Memory-space boundary:** Each memory space gets its own SQLite database file.
3. **Public routing input:** Commands, API requests, MCP tools, and Python-facing helpers accept an optional logical memory-space key, not a raw database path.
4. **Default behavior:** If no memory-space key is provided, use the default memory space.
5. **Safety rule:** User-provided memory-space keys are identifiers only. They must never be interpreted as filesystem paths.
6. **Resolver rule:** One central resolver maps memory-space keys to database files. Public surfaces do not build paths directly.
7. **Compatibility rule:** Legacy `database.path` is not supported in v1.2. Existing installs must move to `database.folder` plus `database.default_memory_space`.
8. **Hermes later:** The future Hermes plugin should always send a resolved profile memory-space key. It should not send database paths.

### Design vocabulary

Use these terms consistently:

- **Memory space:** A logical storage namespace. Examples: `default`, `hermes-default-profile`, `hermes-profile-01HX...`.
- **Memory-space key:** The caller-facing identifier for a memory space.
- **Default memory space:** The memory space used when no key is supplied.
- **Database folder:** The configured directory where Recollectium owns memory-space databases.
- **Resolved database path:** Internal only. The actual SQLite path produced by resolving a memory-space key.

Avoid overloading the existing `space` field. The current memory model already uses `space` for user vs workspace memory. Do not reuse that name for memory-space storage routing.

Recommended names:

- Public CLI flag: `--memory-space`
- Python/API/MCP field: `memory_space_key`
- Config key: `database.folder`
- Default key config: `database.default_memory_space`
- Resolver class: `MemorySpaceResolver`

## 3. Target configuration model

### New config shape

Replace normal use of:

```json
{
  "database": {
    "path": "recollectium.db"
  }
}
```

with:

```json
{
  "database": {
    "folder": "databases",
    "default_memory_space": "default"
  }
}
```

### Resolution rules

- If `database.folder` is relative, resolve it under the XDG data directory.
- If `database.folder` is absolute, use it as provided after normal expansion.
- Ensure the folder exists before opening any memory-space database.
- The default folder should be simple and user-friendly. Recommended default:

```json
{
  "database": {
    "folder": "memory-spaces",
    "default_memory_space": "default"
  }
}
```

The default resolved folder would become something like:

```text
<xdg data dir>/memory-spaces/
```

### Backward compatibility

- `database.path` is rejected with a validation error.
- New configs generated by `recollectium init` should write `database.folder`.

### Config properties to add

In `RecollectiumConfig`, add:

```python
@property
def resolved_database_folder(self) -> Path: ...

@property
def default_memory_space_key(self) -> str: ...

@property
def uses_legacy_database_path(self) -> bool: ...
```

Keep `resolved_database_path` temporarily only for legacy callers and tests, but mark it as compatibility-only in comments and docs.

## 4. Memory-space key validation and path mapping

### Requirements

Memory-space keys must be:

- non-empty strings after trimming
- logical identifiers, not paths
- safe to accept over CLI, API, and MCP
- stable across platforms
- allowed to contain human-readable names within reason
- unable to escape the configured database folder

### Recommended validation

Add a new module:

```text
src/recollectium/memory_spaces.py
```

Core constants:

```python
DEFAULT_MEMORY_SPACE_KEY = "default"
MAX_MEMORY_SPACE_KEY_LENGTH = 128
```

Validation function:

```python
def validate_memory_space_key(value: str | None) -> str:
    """Return normalized memory-space key or raise ValidationError."""
```

Rules:

- `None` is handled by the resolver as default, not by this validator.
- Strip leading and trailing whitespace.
- Reject empty after stripping.
- Reject NUL bytes.
- Reject strings longer than 128 characters.
- Reject path separators `/` and `\\`.
- Reject current-directory and parent-directory semantics such as `.`, `..`, `./x`, `../x`.
- Allow a practical character set such as letters, numbers, dot, underscore, hyphen, and colon.
- If broader Unicode display keys are desired later, store display names separately. Do not allow arbitrary Unicode in the routing key for v1.2.

Recommended regex:

```python
^[A-Za-z0-9][A-Za-z0-9._:-]{0,127}$
```

### Filename mapping

Do not use the key directly as a path component without additional protection.

Recommended deterministic filename:

```text
<slug>--<sha256-prefix>.db
```

Example:

```text
default--37a8eec1ce19.db
hermes-default--9b134ad2ac07.db
```

Where:

- `slug` is a readable sanitized lowercase preview, capped to perhaps 48 characters.
- hash prefix is derived from the exact validated key.
- collisions are practically avoided by the hash.

Also account for SQLite sidecar files:

```text
<name>.db
<name>.db-wal
<name>.db-shm
```

### Optional manifest

For v1.2, strongly consider a small manifest file in the database folder:

```text
memory-spaces.json
```

Purpose:

- map key to resolved filename
- record created time
- record last opened time
- list spaces for status/admin commands
- avoid discovering spaces only by scanning filenames

Example:

```json
{
  "version": 1,
  "spaces": {
    "default": {
      "filename": "default--37a8eec1ce19.db",
      "created_at": "2026-06-19T00:00:00Z",
      "updated_at": "2026-06-19T00:00:00Z"
    }
  }
}
```

If the manifest is skipped for v1.2, status can still resolve the active space, but listing memory spaces becomes less reliable. My recommendation is to include the manifest because it makes diagnostics, backup, and future UI easier.

## 5. New resolver object

Create:

```text
src/recollectium/memory_spaces.py
```

Primary class:

```python
class MemorySpaceResolver:
    def __init__(self, database_folder: Path, default_key: str = "default") -> None: ...

    def resolve(self, memory_space_key: str | None = None) -> MemorySpaceResolution: ...

    def list_spaces(self) -> list[MemorySpaceInfo]: ...
```

Dataclasses:

```python
@dataclass(frozen=True)
class MemorySpaceResolution:
    key: str
    db_path: Path
    is_default: bool

@dataclass(frozen=True)
class MemorySpaceInfo:
    key: str
    db_path: Path
    is_default: bool
    exists: bool
    created_at: str | None = None
    updated_at: str | None = None
```

Resolver invariants:

- Creates the database folder if missing.
- Validates the key or uses default when key is `None`.
- Never returns a path outside the configured database folder.
- Creates or updates the manifest atomically if a manifest is implemented.
- Does not open SQLite itself. It only maps keys to database paths.
- Uses `ValidationError` for invalid keys.

Atomic manifest write:

- Write to a temp file in the same folder.
- Flush and fsync.
- Atomic replace.
- Avoid logging memory content or sensitive local paths unless logging sensitivity allows it.

## 6. RecollectiumCore API design

### Recommended Core construction

Change `RecollectiumCore` from one-store-per-instance to resolver-backed store selection.

Current:

```python
core = RecollectiumCore(db_path=None)
core.store = SQLiteMemoryStore(selected_path)
```

Target:

```python
core = RecollectiumCore(memory_space_key=None)
core.memory_spaces = MemorySpaceResolver(...)
core.store = core._store_for(memory_space_key)
```

But avoid forcing one `RecollectiumCore` instance per memory space. For service and MCP, a single Core instance should serve multiple memory spaces. That means store selection must happen per operation.

Recommended shape:

```python
class RecollectiumCore:
    def __init__(..., memory_space_key: str | None = None, db_path: Path | str | None = None):
        self.default_memory_space_key = memory_space_key or config.default_memory_space_key
        self.memory_space_resolver = MemorySpaceResolver(...)
        self._stores: dict[str, SQLiteMemoryStore] = {}
```

Add helper:

```python
def _store_for(self, memory_space_key: str | None = None) -> tuple[SQLiteMemoryStore, MemorySpaceResolution]:
    resolution = self.memory_space_resolver.resolve(memory_space_key)
    store = self._stores.get(resolution.key)
    if store is None:
        store = SQLiteMemoryStore(resolution.db_path)
        self._stores[resolution.key] = store
    return store, resolution
```

Each operation should resolve its store at the top.

Example target signature:

```python
def add_memory(
    self,
    space: str,
    type: str,
    content: str,
    *,
    memory_space_key: str | None = None,
    workspace_uid: str | None = None,
    metadata: dict[str, object] | None = None,
    source: str | None = None,
    confidence: float | None = None,
    sensitivity: str | None = None,
) -> Memory:
```

Then:

```python
store, resolution = self._store_for(memory_space_key)
...
inserted = store.insert_memory(...)
```

Do this for every operation that reads or writes memory data:

- `add_memory`
- `search_user_memories`
- `search_workspace_memories`
- `list_memories`
- `get_memory`
- `update_memory`
- `archive_memory`
- workspace alias operations
- workspace list/resolve/rename operations
- embedding status/jobs/refresh/maintenance operations
- dev eval and seeded dev operations where appropriate
- database status

### Compatibility constructor

`db_path` exists today. The plan should not remove it blindly in the first commit because many tests and internal helpers may use it.

Recommended v1.2 compatibility:

- Keep `db_path` as a deprecated constructor-only escape hatch for tests/admin code.
- If `db_path` is provided, Core can create a resolver in single-space compatibility mode with a synthetic default key, or directly use a store for that path.
- Public CLI/API/MCP should not expose `db_path`.
- Docs should say `db_path` is internal compatibility and not part of normal user-facing routing.

Long-term v1.3 or v2 can remove `db_path` from public Python API if desired.

## 10. Python API compatibility

### Public Core methods

All Core methods that operate on memory should accept keyword-only `memory_space_key: str | None = None`.

Avoid positional placement to prevent accidental argument shifts.

Good:

```python
def search_user_memories(
    self,
    query: str,
    limit: int = 20,
    *,
    memory_space_key: str | None = None,
    include_archived: bool = False,
    ...
) -> list[SearchResult]:
```

Bad:

```python
def search_user_memories(self, query, memory_space_key=None, limit=20):
```

### Store exposure

If external code uses `core.store`, this change weakens that assumption. Keep `core.store` temporarily as the default memory-space store for compatibility if needed:

```python
@property
def store(self) -> SQLiteMemoryStore:
    store, _ = self._store_for(None)
    return store
```

Prefer updating internal Core code to not depend on `self.store` directly.

## 11. Migration and upgrade plan

### Migration cases

Case A: New install

- Config defaults include `database.folder` and `database.default_memory_space`.
- No legacy handling needed.
- First operation opens default memory-space database under the folder.

Case B: Unsupported legacy config key

The config is invalid and should be rejected with a validation error that tells the operator to use `database.folder` and `database.default_memory_space` instead.

Case C: Unsupported legacy config key

Same as case B: reject it and tell the operator to migrate to `database.folder`.

Case D: Unsupported legacy config key and database.folder together

- Reject the config and instruct the user to remove the legacy key.

- No migration is needed; the config is either valid or rejected.

### Where migration lives

This is a config/storage-layout migration, not a SQLite schema migration.

Do not put it under:

```text
src/recollectium/migrations/versions/
```

That directory is for per-database SQLite schema migrations.

Recommended implementation locations:

- config normalization in `src/recollectium/config.py`
- explicit upgrade behavior in `src/recollectium/update.py` or CLI upgrade flow if existing conventions fit
- helper functions in `src/recollectium/memory_spaces.py`

### Tests

Add tests under:

```text
tests/test_config.py
tests/test_update.py
tests/test_storage.py
```

Test:

- legacy relative path migration
- legacy absolute path migration
- both keys present
- idempotent rerun
- sidecar handling where applicable
- no overwrite of existing default memory-space DB

## 12. Logging and security

### Logging requirements

Add structured logs for:

- memory-space resolution failure
- memory-space database open
- migration from legacy path
- invalid key validation failure, without logging sensitive payloads

Avoid logging memory content or metadata.

Log context can include:

```json
{
  "memory_space_key": "default",
  "is_default_memory_space": true
}
```

Be careful with database paths. Existing logs include `db_path`; with this update, keep paths in debug or redacted sensitivity where appropriate.

### Security requirements

Memory-space keys must not allow:

- path traversal
- absolute path injection
- alternate drive roots on Windows
- SQLite file creation outside the configured folder
- symlink escape if the folder contains malicious symlinks

Implementation should verify final resolved DB path is within the resolved database folder:

```python
resolved_path.resolve().is_relative_to(database_folder.resolve())
```

For Python versions where `is_relative_to` behavior is available, use it. Otherwise implement equivalent parent traversal check.

## 15. Acceptance criteria

The update is complete when all of these are true:

- New installs use `database.folder` and `database.default_memory_space`.
- Existing installs must migrate off legacy path routing before upgrade.
- Every normal operation uses a memory-space key or defaults to the default memory space.
- No normal CLI/API/MCP memory operation accepts a raw database path.
- Explicit memory-space keys isolate data from the default space and from each other.
- User memory and workspace memory semantics remain unchanged inside each memory-space DB.
- Workspace aliases are scoped per memory-space DB.
- Embedding jobs and maintenance operate per selected memory-space DB.
- Invalid keys fail safely and cannot write outside the configured database folder.
- CLI, API, MCP, and Python Core behavior are aligned.
- Docs, OpenAPI, changelog, roadmap, and security guidance are updated.
- Full quality gates pass.

## 16. Future Hermes plugin implication

After v1.2 lands, the Hermes memory provider should do this:

1. On provider initialization, resolve or create a stable Hermes profile memory-space key inside the active Hermes profile home.
2. Store that key in provider state.
3. Include `memory_space_key` on every Recollectium API, MCP, or Python client request.
4. Never send a raw database path.
5. Expose active memory-space key in provider status.

The Hermes plugin does not need Recollectium Core to understand Hermes profiles. Recollectium only needs to understand memory-space keys.

## 17. Risks and mitigations

### Risk: Confusing old `space` terminology

Mitigation: Use `memory_space_key` everywhere for storage routing. Keep `space` only for existing `user` vs `workspace` memory classification.

### Risk: Breaking existing Python users who pass `db_path`

Mitigation: Keep `db_path` as a deprecated compatibility constructor argument for v1.2. Remove from user-facing docs except migration notes.

### Risk: API response shape churn

Mitigation: Add memory-space metadata consistently, or gate detailed metadata behind verbose response mode if compact compatibility is critical. Update OpenAPI and tests either way.

### Risk: Path traversal or unsafe filenames

Mitigation: Validate keys, hash filenames, and verify resolved paths stay under the configured database folder.

### Risk: Migration accidentally overwrites a new default DB

Mitigation: Never overwrite an existing resolved default DB. If both old and new DB exist, report already migrated and leave old DB untouched.

### Risk: Dev seeded DB behavior becomes confusing

Mitigation: Treat seeded dev database as a development memory space or keep it as a development-only explicit path escape hatch. Document and test the chosen behavior. Recommended: map seeded development DB through the resolver as a special development default while `development.use_seeded_database` is true.
